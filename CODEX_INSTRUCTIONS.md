Before performing any task, always read these files:

1. PROJECT_CONTEXT.md
2. CODING_RULES.md
3. MENTAL_MODEL.md

These documents define:

- project purpose
- coding standards
- design philosophy
- product narrative

# Development Workflow

For every request, follow this workflow strictly.

### Step 1 — Context Analysis
Analyze the repository and explain:

- current project structure
- what is already implemented
- what phase the project is currently in
- how the implementation compares to the mental model

Do **not write code yet**.

---

### Step 2 — Premium Design Review

Explain what the current phase must achieve.

Describe:

- what premium quality means for this phase
- how the result should align with the PDF design reference
- how it should reflect the enterprise positioning of the portfolio
- what mistakes would make the result feel cheap or template-like

---

### Step 3 — Gap Analysis

Compare the repository against:

- PROJECT_CONTEXT.md
- CODING_RULES.md
- MENTAL_MODEL.md

Explain clearly:

- what is already aligned
- what is missing
- what must be preserved
- what must not change
- what should be implemented in this phase only
- what should wait for later phases

---

### Step 4 — Implementation Plan

Provide a clear plan including:

- files to create
- files to modify
- component responsibilities
- styling decisions
- layout structure
- responsiveness considerations

Do **not implement yet**.

Stop and wait for approval.

---

### Step 5 — Implementation

After approval:

Implement the plan exactly.

Rules:

- do not install unnecessary libraries
- keep components small and modular
- follow CODING_RULES.md
- keep the design calm and premium
- avoid flashy animation
- avoid unnecessary files

---

### Step 6 — Post-Implementation Self Review

After implementation, review the result and explain:

- whether it aligns with the design philosophy
- whether it matches the mental model
- whether it preserves the enterprise tone
- what could still be improved

---

# Design Expectations

The website must feel like:

- a premium digital CV
- a SaaS product landing page
- a technical sales briefing

The design style must be:

- calm
- technical
- structured
- premium
- enterprise-focused

Avoid:

- flashy animations
- loud gradients
- cluttered layouts
- developer portfolio tropes

---

# Implementation Rules

Always prefer:

- modular components
- clean architecture
- clear spacing
- readable typography
- restrained interactions

Do not overbuild the system early.

Build **one section at a time** following the project phases.

---

# Quality Standard

The final site should leave this impression:

"This person understands both technology and enterprise software adoption."

The goal is **credibility and clarity**, not decoration.
